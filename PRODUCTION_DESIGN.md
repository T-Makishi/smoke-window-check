# 実運用版 設計書

## 1. 結論

このアプリは、公開画面だけをGitHubで管理し、個人情報と写真・動画はSupabaseへ非公開で保存する。受付通知はサーバー処理から担当者メールへ送る。顧客画面に秘密鍵や管理者用データを置かない。

## 2. 現在地

現在の実装は画面・問診・概算・端末内保存・印刷・メール作成まで動作するプロトタイプである。次の制約があるため、そのままでは正式運用しない。

- 問診は会社のデータベースへ保存されない。
- 写真・動画はメールへ自動添付されない。
- 業者設定の4桁パスコードは誤操作防止であり、本人確認にはならない。
- 顧客がブラウザデータを消すと端末内の履歴も消える。
- 迷惑投稿、通信失敗、二重送信をサーバー側で防止・記録できない。

## 3. 採用構成

| 役割 | 採用技術 | 保存するもの |
| --- | --- | --- |
| ソース管理 | GitHub | HTML、CSS、JavaScript、SQL、サーバー処理 |
| 顧客向け画面 | 静的ホスティング | 公開画面のみ |
| 受付API | Supabase Edge Functions | 入力検証、登録、通知処理 |
| 案件データ | Supabase Postgres | 問診、連絡先、概算、受付状態 |
| 添付ファイル | Supabase Storageの非公開バケット | 写真・動画 |
| 業者ログイン | Supabase Auth | 許可された管理者アカウント |
| メール通知 | トランザクションメールサービス | 担当者への新規受付通知 |

メール通知先の初期試験値は `makishi0520@gmail.com` とする。本番移行時は会社管理の受付専用アドレスへ変更する。

## 4. 送信フロー

1. 顧客が問診を入力し、端末上で内容を確認する。
2. 送信ボタンで受付APIへ問診を渡す。
3. APIが必須項目、文字数、ファイル形式、件数、重複送信を検査する。
4. APIが受付番号を発行し、案件をデータベースへ保存する。
5. 写真・動画は案件ごとの非公開フォルダへ保存する。
6. APIが担当者へ受付メールを送る。メール本文には問診概要と受付番号を記載する。
7. 顧客画面に受付完了と受付番号を表示する。
8. 業者は管理画面へログインし、案件と期限付きの添付ファイルURLを確認する。

通知メールだけを正式な原本にしない。メール障害時にも案件を確認できるよう、データベースを受付記録の原本とする。

## 5. データ設計

### `inquiries`

- `id`: 内部UUID
- `reference_number`: 顧客へ表示する受付番号、一意
- `status`: `received`、`reviewing`、`contacted`、`closed`
- `customer_type`: 依頼者区分
- `customer`: 依頼者情報JSON
- `site`: 現場情報JSON
- `diagnosis`: 症状・問診JSON
- `estimate`: 概算JSON
- `inspection_fee`: 現地調査費の同意内容JSON
- `consent_version`: 同意文の版
- `submitted_at`: 受付日時
- `notification_sent_at`: 通知成功日時。未送信なら空

### `inquiry_media`

- `id`: 内部UUID
- `inquiry_id`: 案件UUID
- `storage_path`: 非公開バケット内のパス
- `original_name`: 顧客端末のファイル名
- `content_type`: MIMEタイプ
- `size_bytes`: ファイル容量
- `created_at`: 保存日時

## 6. 権限設計

- 顧客は案件一覧を取得できない。
- 顧客は受付APIを通して新規送信だけできる。
- 業者はログイン後、自社の案件だけ閲覧・状態変更できる。
- Storageバケットは非公開とし、業者画面では短時間だけ有効なURLを発行する。
- Supabaseの秘密鍵とメールサービスのAPIキーはEdge Functionの秘密情報として保管する。
- GitHubへ `.env`、秘密鍵、実在顧客データをコミットしない。

## 7. 入力・障害対策

- 送信ボタンを処理中は無効化し、二重送信を防ぐ。
- APIでブラウザ側と同じ必須検証を再実行する。
- 許可するファイル形式と容量は、契約プランを確認してから確定する。
- 送信に失敗した場合は入力を端末に残し、再送できるようにする。
- 受付登録とメール通知を別状態で記録し、通知失敗を再処理できるようにする。
- 公開前に迷惑投稿対策、利用規約、個人情報の利用目的、保存期間、削除手順を確定する。

## 8. LINEの扱い

初期運用の正式経路はメール通知とする。LINEの共有画面は顧客が使う補助機能として残すが、宛先を保証できないため正式受付完了とは扱わない。LINE公式アカウントとの自動連携は、メール受付が安定した後に別工程で実装する。

## 9. リリース条件

- 自動テストが成功する。
- iPhoneとAndroidの実機で入力、撮影、送信、再送を確認する。
- 受付メールとデータベースの内容が一致する。
- 添付ファイルを一般公開URLから取得できない。
- 業者設定と案件一覧へ未ログインで入れない。
- 印刷、複数症状、端末内下書きが引き続き動作する。
- 個人情報の同意文と問い合わせ先を実際の会社情報に更新する。
